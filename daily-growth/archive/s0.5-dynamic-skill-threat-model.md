# S0.5 — Threat model for dynamic (composition) skills
**Spike:** Phase 0 · **Date:** 2026-06-20 · **Status:** design note (gates Phase 1)
**Question:** What can a malicious or buggy runtime skill do, which existing guards cover it, and what must Phase 1 add before any dynamic skill is allowed to execute?

## Scope
A **composition skill** = a named, ordered list of *existing native commands*, defined as a
`<workspace>/skills/*.md` file, discovered at runtime and resolved in `Commands.call/2` after the
catalog miss (see S0.6). This is the **only** handler kind in scope. Script (kind 2) and
arbitrary-code (kind 3) handlers are **out of scope and must be rejected** until a follow-up threat
model clears them. The asset we protect: the user's Google Workspace + filesystem reachable through
the 8 gated commands, plus run cost (tokens) and the audit feed's integrity.

## Trust premise
Two facts make composition skills *mostly* safe by construction:
1. A composition skill can only invoke commands that **already exist** — it adds no new raw
   capability, only new *sequencing*.
2. If every step re-enters `Commands.call/2` with the **invoker's** caller class, the skill can never
   exceed the trust of whoever invoked it. The existing `authorize/2` gate then applies per step.

The threats below are the ways that premise can be violated or sidestepped.

## Threats

| # | Threat | Existing guard | Verdict | Phase-1 mitigation required |
|---|---|---|---|---|
| T1 | **Privilege escalation** — skill executor calls handlers via `apply/3` directly, skipping `authorize/2`, so a skill reaches gated commands. | `authorize/2` (only if traversed) | **OPEN by design risk** | **Hard rule: skill steps dispatch through `Commands.call/2`, never `apply/3`.** One choke point; covered by tests asserting a skill cannot reach a gated cmd as `:agent_untrusted`. |
| T2 | **Trust laundering** — skill declared/run at a higher caller class than its invoker. | caller model | PARTIAL | Effective caller = **min(skill-declared tier, invoker caller)**; default to most-restricted. Skill never raises trust. |
| T3 | **Gated-command laundering** — chaining safe steps to reach `gmail_send`/`*_delete`. | `authorize/2` refuses gated for `:agent_untrusted` (per step) | COVERED | None new — *provided* T1 holds. Verify with a test. |
| T4 | **Resource exhaustion / loops** — skill loops or fans many non-gated calls (e.g. `gmail_search` ×1000); cost + API abuse. | wallet budget caps **run cost** only | PARTIAL (gap) | **Per-caller/per-command rate limit (1C)** + a **max-steps cap per skill execution** + reject skills with cyclic/`goto` control flow (composition is a *flat ordered list*, no loops). |
| T5 | **Malicious skill authorship via prompt injection** — an agent processing untrusted content (an email) is steered to write an exfiltrating `skills/*.md`. | files are git-diffable + operator-visible | PARTIAL | Skills are **disabled on creation; never auto-enabled** (Phase 3 proposes→human approves). Writing to `skills/` is itself an **audited (`:skill_invoke`/`:skill_write`) and gated** action. A disabled skill is non-resolvable. |
| T6 | **Policy bypass via resolution order** — skill resolves *after* the catalog, so a check wired only into the static catalog misses it. | — | OPEN | `PolicyEngine.check/2` runs in `call/2` **before** `dispatch/2`, for native **and** composition. Skill expansion re-enters `call/2` per step → re-checked. Never gate at `build_catalog/0`. |
| T7 | **Name shadowing/collision** — a skill named like a native command, or two skills colliding. | catalog-first resolution (native wins) | PARTIAL | On load, **reject** any skill whose name is in `by_name()` (native) or duplicates another skill; require a validated name charset. Surface `source: composition` in the catalog so shadowing is visible. |
| T8 | **Handler-kind escalation** — a skill file declares `handler_kind: script`/`code`. | — | OPEN | **Allowlist: composition only.** Any other `handler_kind` → load-time reject + `:security_block`, not best-effort execution. |
| T9 | **SSRF / arg injection through a step** — a step takes a URL/path arg. | `UrlGuard.req_step/1` in the command handlers | COVERED | None new — inherited *because* steps go through real command handlers (T1 rule again). Don't let skills pass raw opts that bypass per-command arg validation. |
| T10 | **Audit-chain gaps** — skill runs but the feed shows only the leaf commands, losing "which skill did this." | `Sentinel.observe` | PARTIAL | Emit a `:skill_invoke` event at expansion; tag each step's existing `:command_invoke` meta with `skill: <name>` provenance. |

## The execution gate (minimum bar before any dynamic skill runs)
A skill is resolvable/executable **only if all hold** — checked at load and at dispatch:
1. `enabled: true` (operator-approved; default false). [T5]
2. `handler_kind == composition`. [T8]
3. Name is unique and does not collide with any native command. [T7]
4. Body is a flat, finite ordered list of known native command names (no loops/recursion);
   length ≤ a configured `max_steps`. [T4]
5. Every step is dispatched through `Commands.call/2` with `caller = min(skill_tier, invoker)`. [T1, T2]
6. `PolicyEngine.check/2` passes for the skill **and** for each expanded step, before dispatch. [T6]
7. Per-caller/per-command rate budget not exhausted. [T4]
8. `:skill_invoke` recorded with provenance; each step audited. [T10]

If any check fails: refuse with `{:error, :requires_confirmation}` (or `:policy_blocked`), record a
`:security_block`, do not execute.

## What Phase 1 must build (derived)
- **Enforcement:** generalize `authorize/2` into `PolicyEngine.check/2` at the `call/2` choke point
  (declarative rules in `<workspace>/memory/policy.md`).
- **Limits:** per-caller/per-command rate counter + per-skill `max_steps`.
- **Loader guards:** `enabled` default-false, handler-kind allowlist, name-collision rejection,
  flat-list validation.
- **Audit:** `:skill_invoke` category + step provenance.
- **Out of scope (defer behind their own threat model):** script and arbitrary-code handler kinds,
  cross-machine/A2A skill sharing, self-*enabling* skills.

## Residual risks accepted for v1
- A *trusted* caller can already do anything a skill can — composition adds no power there; the model
  protects the `:agent_untrusted`/`:agent`/`:mcp` paths, which is where untrusted content flows.
- DNS-rebinding/TOCTOU in `UrlGuard` (pre-existing, noted in `url_guard.ex`) is unchanged by skills.
- Token-cost abuse is bounded by the wallet ledger at the *run* level even before 1C lands; 1C
  tightens it to the *call* level.
