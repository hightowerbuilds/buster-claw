# S0.4 — Kimi Agent Swarm coordination → an OTP fan-out/fan-in contract
**Spike:** Phase 0 · **Date:** 2026-06-20 · **Status:** design note (feeds Phase 4)
**Question:** The fan-out/fan-in contract + failure handling to mirror in OTP.

> Honesty note. My knowledge cutoff is Jan 2026; Kimi K2.6's "Agent Swarm" is dated ~Apr 2026, so I can't verify its internals — only what vendor/secondary pages assert. The originating essay (`lobsterattack.yachts`) is an unreachable SPA. Claims below are tagged **[verified]** (corroborated multi-agent pattern in-cutoff), **[marketing]** (vendor/secondary claim, unconfirmed), **[inferred]** (my reasoning). The "300 agents / 4,000 steps" numbers are **[marketing]** — take the *pattern*, not the figures.

## Findings

- **Decomposition model: heterogeneous, coordinator-driven (supervisor-worker).** K2.6 acts as an "adaptive coordinator": it decomposes a task into *heterogeneous* subtasks (search vs. deep-research vs. analysis vs. writing) and "dynamically matches tasks to agents based on their skill profiles and available tools." **[marketing]** This is the classic **supervisor-worker / scatter-gather** pattern — a coordinator splits, dispatches to role-typed workers, gathers. **[verified]** (Note: this is *not* uniform map-reduce — workers are role-specialized, not identical mappers. **[inferred]**)
- **Scale figures are a ceiling, not a contract.** "300 sub-agents, 4,000 coordinated steps" is a *total* step budget (~13 steps/agent avg), and even the vendor pages concede real API limits are "50–100 concurrent with queue-based throttling." **[marketing]** The honest takeaway: a **bounded concurrency cap + a shared step/cost budget** is the actual mechanism; the headline number is the marketing skin. **[inferred]**
- **Fan-in is undocumented.** Secondary analysis (Verdent) flags that the aggregation/conflict-resolution mechanism is *not published*; sub-agents can hold "inconsistent views of shared state" and the auto-resolution "is not auditable." **[marketing]** This is the gap Buster Claw must close explicitly: **make fan-in a deterministic, audited step, not an opaque merge.** **[inferred]**
- **Failure handling is "detect-and-reassign."** K2.6 reportedly "detects when an agent fails or stalls, reassigns or regenerates subtasks," but the detection trigger (timeout / error / validation) is undocumented, and *coordinator* failure is unspecified. **[marketing]** In OTP this is solvable and observable: supervised tasks + timeouts + a typed result per worker. **[verified]**
- **Maps cleanly onto in-cutoff primitives.** Scatter-gather over a bounded pool = `Task.async_stream` with `max_concurrency`; per-worker isolation/restart = `Task.Supervisor`; LangGraph-style fan-out and Erlang process pools are the same shape. **[verified]**

## The fan-out/fan-in contract for Buster Claw

A queue item that opts into fan-out carries a `plan`: `{role, prompt, budget_cents}` triples (the coordinator step that produces this plan is itself one `AgentRunner` run — keep the planner serial). The contract:

1. **Bounded concurrency.** Hard cap `max_concurrency` (start at **3–4**, config `:swarm_max_concurrency`). N sub-runs in the plan may exceed the cap; they queue. Each sub-run is one `AgentRunner.run/2` Port (unchanged), so the existing per-run wall-clock timeout (`run_timeout_ms`) and trusted/untrusted token selection apply *per sub-run*.
2. **Provenance to Sentinel, per sub-run.** Every sub-run emits its own `:command_invoke` observation tagged `{swarm_id, role, sub_run_index, exit_status, duration_ms}` — not one blurred event for the whole swarm. Inherit the queue's fail-closed provenance: if *any* parent item is untrusted, **all** sub-runs get the agent (untrusted) token. Don't let fan-out launder an untrusted item into a trusted sub-role.
3. **Per-role budget draw from the wallet ledger.** Before a sub-run starts, reserve its `budget_cents` against the swarm's budget (a single `wallet_transactions` expense row per sub-run keyed by `{swarm_id, role}`, debiting the shift's budget envelope). **Reserve up front, reconcile on completion** to actual token cost. The whole swarm draws against the *same* per-shift cap the Dispatcher already enforces (`max_runs_per_shift`) — count each sub-run as a run so fan-out can't bypass the run-cap budget brake.
4. **Partial-failure aggregation.** Fan-in is an explicit, deterministic step. Collect a typed result per sub-run: `{:ok, output}` / `{:error, reason}` / `{:timeout}`. The aggregator (a final, serial `AgentRunner` step, or pure Elixir for trivial merges) receives **all** results including failures, and the queue item resolves as: `done` if successes ≥ a required `quorum`, else `block` with a note listing the failed roles. Never silently drop a failed sub-run.
5. **k-of-n policy is per-item.** Default `quorum = n` (all must succeed) for irreversible work; allow `quorum = ceil(n/2)` or a named-required-roles set for research-style fan-out where partial coverage is useful. When `k` of `n` fail and `k > n - quorum`, the item **blocks** (not done), preserving "halt cleanly for the operator" semantics.

## OTP primitive recommendation

**Use `Task.Supervisor.async_stream_nolink/4` with `max_concurrency`** (a `Task.Supervisor` started under the app supervision tree), **not** a raw `DynamicSupervisor` and not bare `Task.async_stream`.

- `async_stream` gives **bounded concurrency for free** (`max_concurrency`) — exactly the cap that matters; the "300 agents" headline collapses to this knob. It also gives a built-in `:timeout` and `on_timeout: :kill_task` per element, and preserves result order for deterministic fan-in.
- `Task.Supervisor` + `_nolink` means **one sub-run crashing does not take down the coordinator** — you get `{:exit, reason}` as that element's result instead of a propagated kill. That is the partial-failure contract, expressed in the supervisor.
- `DynamicSupervisor` is the right tool only if sub-runs must outlive the coordinator call or be individually addressable/restarted — we don't need either; sub-runs are bounded, scatter-gathered, and ephemeral. Reach for it later only if a role needs independent retry/restart lifecycle. **[inferred]**

Sketch:

```elixir
plan
|> Task.Supervisor.async_stream_nolink(BusterClaw.SwarmTaskSup, &run_sub/1,
     max_concurrency: max, timeout: run_timeout_ms, on_timeout: :kill_task,
     ordered: true)
|> Enum.map(&normalize_result/1)   # {:ok, r} | {:exit, why} | {:exit, :timeout}
|> aggregate(quorum)               # -> :done | {:block, failed_roles}
```

## Failure handling + crash-loop composition

- **Coordinator failure (the gap Kimi leaves open):** the swarm coordinator is just another monitored process under the Dispatcher. If it dies mid-swarm, the Dispatcher's existing `{:DOWN, ...}` path resets, and any queue item it claimed is returned by `reclaim_orphans/0` on boot — same crash-safety the serial pump already has. In-flight sub-run Ports are killed by `on_timeout/async_stream` teardown; their reserved budget is reconciled (released) on the next reconcile pass.
- **Crash-loop brake composes by counting the *swarm*, not each sub-run.** A swarm that fails aggregation (blocks) is **one** failed Dispatcher tick, so the existing `dispatched_count` / `max_runs_per_shift` budget brake and the Orchestrator's "N consecutive tick failures → stop shift + `:security_block`" brake both still fire correctly. **Do not** let one flaky sub-role increment the crash-loop counter — otherwise fan-out makes the brake trip-happy. A sub-run failure is data fed to the aggregator; only an *unrecoverable swarm* (coordinator crash, or quorum-block streak) is a tick failure.
- **Budget brake still front-runs the run:** keep the existing within-budget check before launching a swarm, and treat the swarm's total reserved `budget_cents` as the gate — a fan-out can request more than a serial run, so check the *sum* against the per-shift cap, fail-closed (block + Sentinel `:security_block`) if it would exceed.

Net: mirror Kimi's *pattern* (coordinator splits → bounded role-typed workers → explicit fan-in), but replace its undocumented/opaque parts (stall detection, conflict-resolution merge, coordinator-failure behavior) with OTP's observable equivalents (supervised `async_stream`, typed per-element results, audited per-sub-run Sentinel provenance, ledger-backed budget reservation).
