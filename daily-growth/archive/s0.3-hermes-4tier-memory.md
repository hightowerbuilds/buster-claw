# S0.3 — Hermes 4-tier memory, mapped to Buster Claw stores
**Spike:** Phase 0 · **Date:** 2026-06-20 · **Status:** design note (feeds Phase 2)
**Question:** The four tiers; map each to a store; which do we need for v1?

## Findings

**Honesty note.** My knowledge cutoff is Jan 2026 and Hermes Agent's memory write-ups
post-date it; the canonical source essay (lobsterattack.yachts) is an unreachable SPA. So
Hermes specifics below are reconstructed from secondary web sources, not the primary spec.
Where the "four tiers" can't be pinned to one authoritative list, I anchor the mapping on
**MemGPT/Letta**, whose tiers are in-cutoff and verifiable, and say so.

- **[inferred] There is no single canonical "Hermes 4-tier" list — the sources disagree.**
  The cleanest claim is bounded, curated, always-on memory split across stores rather than
  one flat blob. Two distinct "four-ish" framings appear:
  - A *proposed* tiered-storage redesign (GitHub issue #32726) with **Layer 0 root index
    (`~/.hermes/index.json`) → Layer 1 project data files → Layer 2 skills → Layer 3 memory
    (rules + a pointer to the index) → Layer 4 `session_search` (FTS5 fallback)**. That's a
    proposal (5 named layers), `[marketing]`/`[inferred]`, not shipped behavior.
  - A "closed learning loop of four elements" (i-scoop): persistent memory → search →
    skill creation → skill refinement. That's a *loop*, not a storage hierarchy. `[marketing]`
- **[inferred] What the secondary sources *do* agree on (the real, shipped shape):**
  1. **Built-in core memory** — `MEMORY.md` (agent notes, ~2.2k chars) + `USER.md` (user
     profile, ~1.4k chars), markdown in `~/.hermes/memories/`, frozen into the system prompt
     at session start. Always-on, bounded, curated.
  2. **Session-history search** — every CLI/chat session in SQLite `~/.hermes/state.db`,
     queried via `session_search` over an **FTS5** index (returns real messages, no LLM
     summarization, no truncation). `[inferred, well-supported across 3 sources]`
  3. **(Optional) external memory provider** — pluggable semantic backend (Honcho/Mem0/etc.).
- **[verified] The real, in-cutoff analog is MemGPT → Letta.** MemGPT (arXiv 2310.08560)
  splits **main context (in-prompt "RAM")** vs **external context ("disk")**. Letta makes it
  three tiers: **core memory** (in-context, agent-editable) / **recall memory** (searchable
  past conversation, disk-cache) / **archival memory** (long-term store queried by tool call).
- **[inferred] The honest "four-tier" model** = MemGPT's three + a **user/identity model**
  split out (Hermes literally separates `USER.md` from `MEMORY.md`; generative-agents and
  most assistant stacks treat the user profile as its own durable tier). That four-tier set —
  **(1) ephemeral working context, (2) durable agent notes, (3) user model, (4) searchable
  history** — is the best-supported synthesis, and it maps cleanly onto our stores.

## Tier → Buster Claw store mapping

| Tier (synthesized 4-tier) | Hermes / MemGPT analog | Buster Claw store | Status |
|---|---|---|---|
| **1. Ephemeral run context** (in-prompt working memory; lives & dies with one run) | MemGPT main context; Letta core; Hermes session window | The agent's live prompt + per-conversation transcript in SQLite **`agent_chat_messages`** (`conv_id`, `role`, `content`, `session_id`, `cost_usd`, `num_turns`) | **exists** |
| **2. Searchable history / run recall** (find what past runs did) | Letta recall; Hermes `session_search` (FTS5) | Planned **`run_summaries`** table (written by `Dispatcher.record_outcome/3`) + SQLite **FTS5** index + **`memory_search`** command | **planned (Phase 2)** |
| **3. Durable agent notes / learned facts** (curated, cross-run, agent-editable) | MEMORY.md; Letta archival | **`<workspace>/memory/*.md`** — today only scaffolding (sole runtime write: `trusted-email-senders.md`; the DB `memories` table was dropped 2026-06-14) | **scaffold only** |
| **4. User / identity model** (stable prefs, who the user is) | Hermes USER.md; generative-agents profile | Also **`<workspace>/memory/*.md`** (a dedicated `user.md` / prefs file) — no separate store today | **not built** |

## Minimal set for v1

A solo-dev, file-first app should **not** build all four. Recommended minimal set is **Tier 1 + Tier 2**:

- **Tier 1 (ephemeral run context) — keep, already done.** `agent_chat_messages` is the live
  transcript; this is free and non-negotiable. Buys: coherent in-run reasoning.
- **Tier 2 (`run_summaries` + FTS5 + `memory_search`) — build this; it's the whole point of
  Phase 2.** Buys cross-run recall ("what did the agent decide last Tuesday") without
  embeddings, with one cheap SQLite index. This is the highest-leverage tier and exactly what
  Hermes's `session_search` and Letta's recall tier prove out. **Highest priority.**

**Cut / defer for v1:**

- **Tier 3 (durable `memory/*.md` notes) — DEFER, thin version only.** `memory/` is already
  just scaffolding and a DB table for it was *dropped as unused* — that's the codebase telling
  you flat agent-notes memory wasn't earning its keep. Don't rebuild it as a tier. If anything,
  let the agent append to one markdown file via normal file tools (no schema, no command).
  Reason to cut: Tier 2 search over real run history beats hand-curated notes that rot, and
  you avoid the curation/eviction logic that makes MemGPT-style memory expensive.
- **Tier 4 (user model) — CUT for v1.** One solo user who *is* the developer doesn't need a
  modeled profile; `trusted-email-senders.md` already covers the one real "about the user"
  fact. Revisit only if Buster Claw gets multiple users or long-horizon personalization.

**Net:** ship Tiers 1+2. Tier 1 is done; Tier 2 (`run_summaries` + FTS5 + `memory_search`) is
the build. Treat Tiers 3 & 4 as a single optional markdown file the agent can edit with
existing file tools — not as engineered memory tiers.
