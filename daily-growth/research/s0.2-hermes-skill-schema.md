# S0.2 — Hermes skill files + a chosen skill-file schema
**Spike:** Phase 0 · **Date:** 2026-06-20 · **Status:** design note (feeds Phase 1A)
**Question:** Minimal portable skill-file schema to adopt verbatim.

---

## Findings

### The skill-file schema everyone converges on (Anthropic / agentskills.io)

The real, verifiable anchor is the **Agent Skills / SKILL.md** standard. Both Anthropic
(`anthropics/skills`) and the open `agentskills.io` spec describe the same shape: a directory
named after the skill, containing a `SKILL.md` with YAML frontmatter + a Markdown body.

The frontmatter is deliberately tiny — **only `name` and `description` are required** **[verified]**
(agentskills.io/specification):

| Field           | Required | Constraint **[verified]** |
| --------------- | -------- | ------------------------- |
| `name`          | yes | ≤64 chars; `[a-z0-9-]`, no leading/trailing/consecutive hyphens; **must match the parent dir name** |
| `description`   | yes | ≤1024 chars; what it does *and when to use it* (this is the activation signal) |
| `license`       | no | license name or file ref |
| `compatibility` | no | ≤500 chars; env requirements |
| `metadata`      | no | arbitrary string→string map (this is where `version` lives) |
| `allowed-tools` | no | space-separated, pre-approved tools (**experimental**) |

Body: free-form Markdown, recommended <5000 tokens, with **progressive disclosure** — only
`name`+`description` (~100 tokens) load at startup; the full body loads on activation; `scripts/`,
`references/`, `assets/` load on demand **[verified]**. Anthropic's own docs frame frontmatter as
"HOW the skill runs," body as "WHAT to do," and call `description` "the primary signal Claude uses
to decide when to invoke a skill" **[verified]**.

### What makes a skill *portable*

- **Plain Markdown + a 2-field minimum.** Any runtime that can read a file and split YAML
  frontmatter can consume it; the required surface is trivially small. **[verified, inferred]**
- **Description-as-trigger.** Routing lives in `description`, not in a host-specific manifest, so
  the same file routes correctly in any agent. **[verified]**
- **Progressive disclosure + relative file refs.** Keeps the always-loaded cost ~100 tokens and
  avoids absolute paths, so a skill folder is self-contained and movable. **[verified]**
- **`metadata` as the escape hatch.** Anything non-standard (versions, our tier hints) goes in
  `metadata` without breaking spec validators — the cross-runtime portability seam. **[inferred]**

### Hermes Agent specifically

- Hermes treats skills as `SKILL.md` files — "pure text, easily mutated, and directly measurable" —
  stored in a local `skills/` directory in the repo. **[verified]** (NousResearch/hermes-agent-self-evolution README/PLAN)
- Default size budget **15KB per skill** (configurable), injected as prompt text. **[verified]**
- **Hermes does NOT publish a frontmatter schema or a tool-reference syntax.** Both the README and
  PLAN say skills are "procedural instructions" but give no field list and no syntax for embedding
  command/tool calls inside a skill. **[verified — i.e. the absence is confirmed]** So any "ordered
  command steps" structure is *our* invention; it cannot be copied from Hermes. We anchor it on the
  Agent Skills schema (real) and our own `Commands` catalog instead. **[inferred]**
- The originating essay (lobsterattack.yachts) is an unreachable SPA — not used. **[unverifiable]**

## Proposed Buster Claw skill-file schema (with a full example)

**Design rule:** a Buster Claw skill = a named, ordered list of **existing native commands**. We adopt
the Agent Skills frontmatter close to verbatim (`name`, `description`, `metadata`) for portability,
and add a Buster-specific block — `steps` (ordered command refs) + `tier` hint — carried so it
**round-trips through our existing `Frontmatter.split/1` parser**.

**Hard constraint we must respect (from the codebase):** `Frontmatter.split/1`/`parse_value` parses
strings, numbers, booleans, and **JSON-style** `[...]`/`{...}` values — it does **not** parse native
YAML block lists (`- item`). Therefore the ordered steps are encoded as a **single-line JSON array of
objects** in frontmatter (`steps: [{...}, {...}]`), which round-trips losslessly today with zero
parser changes. (A human-readable mirror lives in the body for the agent to read.) Discovery mirrors
`Jobs.list/0`: scan `<workspace>/skills/*.md`, `Frontmatter.split/1`, take `name`/`description`/`steps`.

```markdown
---
name: weekly-finance-digest
description: >-
  Pull recent SEC filings and a quote for a ticker, save a research note to the
  Library, and email it to the operator. Use for a scheduled or requested
  weekly financial summary on a single company.
metadata: {"version":"1.0.0","author":"luke"}
tier: restricted
args: {"symbol":{"type":"string","required":true},"to":{"type":"string","required":true}}
steps: [{"command":"finance_filings","args":{"symbol":"$symbol"}},{"command":"finance_quote","args":{"symbol":"$symbol"}},{"command":"document_save","args":{"name":"$symbol weekly digest","body":"$prior"}},{"command":"gmail_send","args":{"to":"$to","subject":"$symbol weekly digest","body":"$prior","confirm_send":true}}]
---

# Weekly finance digest

Produce a one-company weekly digest end to end.

1. `finance_filings {symbol}` — newest SEC filings.
2. `finance_quote {symbol}` — latest quote (carries as-of).
3. `document_save` — write the digest note to the Library.
4. `gmail_send` — email it to `{to}` (outbound; gated — needs approval on
   untrusted-origin runs).

Notes: `$symbol`/`$to` are skill args; `$prior` is the previous step's result.
```

**Frontmatter field reference (the part to adopt verbatim):**

| Field | Source | Meaning |
| ----- | ------ | ------- |
| `name` | Agent Skills (verbatim) | filename stem == name; `[a-z0-9-]` |
| `description` | Agent Skills (verbatim) | what + when (routing signal) |
| `metadata` | Agent Skills (verbatim) | JSON map; holds `version` (spec puts version here, not top-level) |
| `tier` | Buster Claw | `safe` \| `restricted` — the **declared** ceiling; Phase 1A must still authorize each step through `Commands.call/3` so the real `tier`/`gated` rules win |
| `args` | Buster Claw | JSON map mirroring our `Commands` args schema |
| `steps` | Buster Claw | **ordered** JSON array of `{command, args}`; `command` MUST be a real `Commands` catalog name |

**Why this round-trips cleanly:** every value is a scalar or JSON `{}`/`[]` — exactly the shapes
`parse_value` already handles and `Frontmatter.build` already emits. No new parser, no DB table;
`Skills.list/0` is a near-clone of `Jobs.list/0` over a `skills/` subdir. Step authorization reuses
`Commands.command_tier/1`, `command_gated?/1`, and the `:agent_untrusted` caller path verbatim —
so a skill can never escalate past what a raw command call already allows.

**Portability note:** `name`/`description`/`metadata` are spec-clean, so the file validates against
`agentskills` tooling; `tier`/`args`/`steps` are ignorable extras (could move under `metadata` later
if strict portability is ever needed). We get cross-runtime legibility *and* native executability.

## Self-evolution: worth it?

DSPy + GEPA (the `hermes-agent-self-evolution` repo, ICLR 2026 oral, MIT) treats a skill's text as an
optimizable parameter: wrap it as a DSPy module, run it on test tasks, let GEPA read execution
**traces** ("why it failed, not just that it failed") and mutate the prose, keep the Pareto-best
variants. No GPU; ~$2–10/run **[verified]**. The headline +39.5% completion number is **[marketing]**
(single held-out example; not in the README I fetched).

**Pressure-test for a solo-dev Elixir app:** keep it marked *optional later*. The whole loop is a
Python/DSPy harness that operates **on** files — and our skills are already plain files. So the
prerequisite isn't the optimizer; it's (a) a labeled eval set of tasks per skill and (b) a scoring
function. A solo dev with a handful of skills has neither, and hand-editing the Markdown is faster
and free. The genuinely portable lesson to bank now: **make skills pure text and make outcomes
measurable** (log each step's `{:ok,_}`/`{:error,_}` via the Sentinel audit we already write). That
keeps the door open to bolt GEPA on later without redesign, at zero cost today. Verdict: defer; the
file-first schema above is the only commitment Phase 1A needs.
