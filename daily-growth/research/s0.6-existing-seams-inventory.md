# S0.6 — Inventory of existing seams to extend (don't rebuild)
**Spike:** Phase 0 · **Date:** 2026-06-20 · **Status:** design note (gates all phases)
**Question:** Which existing code seams do the roadmap phases extend, so we add to them instead of building parallel machinery?

This note exists because the original roadmap proposed new subsystems (a `skill_definitions`
table, a Sentinel-side enforcer, a budget system) for capabilities that **already have a
seam in the code**. Every phase below should hang off the named seam.

## The five seams

### 1. The dispatch choke point — `Commands.call/2` + `authorize/2`
`lib/buster_claw/commands.ex:1267-1401`. **This is the single place every command passes
through, and it already enforces, not just logs.**
- `call/2` → `authorize(name, caller)` → on `:ok` `dispatch/2`; on refusal records
  `Sentinel.Pending.record/3` and returns `{:error, :requires_confirmation}` (l.1270-1282).
- Caller classes (richer than the roadmap's "3 tiers"): `:trusted`, `:agent_untrusted`,
  `:agent`, `:mcp` (l.1387-1401). `:agent_untrusted` is refused the **gated** set (8 outbound/
  irreversible commands); `:agent`/`:mcp` may run only `:safe`-tier; `:trusted` unrestricted.
- `dispatch/2` resolves via `apply(__MODULE__, String.to_existing_atom(name), [args])` (l.1374-1380)
  after `has_command?/1`.
- **Extend here for:** Phase 1A composition-skill resolution (fall through *after* the catalog
  miss, still inside `call/2`); Phase 1B `PolicyEngine.check/2` (insert *before* `dispatch/2`, so
  it gates native **and** composition uniformly); Phase 1C rate-limit counter.
- **Rule:** composition skills must re-enter `call/2` per step (inherit `caller`), never call
  `apply/3` directly — that's the whole security argument (see S0.5).

### 2. The audit spine — `Sentinel.observe/4`
`lib/buster_claw/sentinel.ex:50-76`. Classifies → redacts → persists (`security_events`) →
PubSub broadcast. `commands.ex` already feeds it `:security_block` (refusals) and
`:command_invoke` (mutate/trigger outcomes) via `record/3` (l.1315-1325, inline under the
test sandbox, `Task`-offloaded in prod).
- **Extend here for:** a new `:skill_invoke` category (skill provenance), and Phase 4 sub-run
  provenance. Sentinel stays the recorder; enforcement lives at seam #1. Do **not** move the
  policy check into `observe` — it fires after the fact and off the caller path.

### 3. The file-discovery pattern — `Jobs.list/0` + `TrustedSenders`
`lib/buster_claw/jobs.ex:25-38` discovers capability units from `<workspace>/job-descriptions/*.md`
at runtime; `trusted_senders.ex` reads `<workspace>/memory/trusted-email-senders.md`, memoized in
`:persistent_term` with safe-empty default. `commands.ex` uses the same `:persistent_term`
memoization for its catalog (`catalog/0`, `by_name/0`).
- **This is the template for `skills/*.md` and `policy.md`** — runtime-discovered, git-diffable,
  operator-editable, memoized, safe-empty default. Phase 1 reuses it verbatim; **no new registry
  table** for skill *definitions*. Use the same memoize-and-invalidate-on-write shape.

### 4. The budget ledger — `wallets` / `wallet_budgets` / `wallet_transactions` + `WalletPoller`
The Dispatcher already consults a budget gate before launching a run (`dispatcher.ex:123-127`).
- **Extend here for:** Phase 1C spend limits and Phase 4 per-role/per-sub-run budget draws.
  Phase 1C's *new* work is only the per-caller/per-command **call-rate** counter — the **spend**
  side already exists. Don't stand up a second budget system.

### 5. The post-run hook — `Dispatcher.record_outcome/3`
`lib/buster_claw/dispatcher.ex:249-275`. Already runs after each headless run: records the Sentinel
outcome + bumps shift counters. `AgentRunner.run/2` (`agent_runner.ex:74-82`) just launches the
Port and returns a result — it is *not* the hook point.
- **Extend here for:** Phase 2 `run_summaries` persistence (write the structured summary right
  beside the existing outcome record).

## Also reuse (don't reinvent)
- **Durable queue:** `Dispatch.claim_next/2` (pessimistic lock + `reclaim_orphans/0`) — Phase 4
  fan-out claims N items / spawns N sub-runs through this, not a new queue.
- **Safety brake:** `Orchestrator.trip_crash_loop/2` (`orchestrator.ex:54-90`) — Phase 4 parallelism
  must compose with this, not bypass it.
- **SSRF:** `UrlGuard.req_step/1` — composition-skill steps that fetch inherit it for free *because*
  they go through the existing command handlers (another reason for the seam-#1 rule).

## One-line phase → seam map
| Phase | Primary seam to extend |
|---|---|
| 1A composition skills | `Commands.call/2` resolution + `Jobs.list/0` file pattern |
| 1B policy engine | `Commands.authorize/2` → `PolicyEngine.check/2` before `dispatch/2` |
| 1C rate limits | `Commands.call/2` counter + the wallet ledger (spend side exists) |
| 2 memory | `Dispatcher.record_outcome/3` (write) + new FTS table + `memory_search` cmd |
| 3 self-improve | reads `security_events` + `run_summaries`; writes `skills/*.md` drafts |
| 4 parallel | `Dispatch.claim_next/2` + `Task.Supervisor` fan-out; `Orchestrator` brake; wallet draws |
