# S0.1 — OpenClaw skill model + A2A gateway
**Spike:** Phase 0 · **Date:** 2026-06-20 · **Status:** design note
**Question:** Skill-manifest shape; adopt A2A for our roles vs. stay on loopback CLI?

## Honesty note
My knowledge cutoff is Jan 2026. **OpenClaw, ClawHub, and the `win4r/openclaw-a2a-gateway` plugin all post-date it** and I could not reach a canonical primary source I trust as a project of record (the lobster SPA never resolves to readable HTML). I fetched the GitHub/docs pages, but those read as a third-party ecosystem + SEO content, so OpenClaw-specific specifics are **[marketing]** unless echoed by a pre-cutoff standard. The one solidly verifiable anchor is the **A2A protocol itself** (`a2a-protocol.org`, Linux Foundation–governed, predates cutoff) — those claims are **[verified]**. Treat the OpenClaw skill schema as "shaped like Anthropic Agent Skills" — that analogy is the real load-bearing fact.

## Findings
- **[verified]** A2A is a real, standards-track protocol (Google → Linux Foundation). Discovery is an **Agent Card** JSON at `/.well-known/agent.json`; payloads are **JSON-RPC 2.0** over HTTP, with SSE for streaming. The card carries identity, endpoint `url`, auth, and a `skills[]` capability list. (a2a-protocol.org/v0.3.0/specification)
- **[verified/analogy]** The "SKILL.md + YAML frontmatter + markdown body, discovered by walking a directory tree" model is exactly Anthropic's **Agent Skills** convention (real, pre-cutoff). OpenClaw's docs describe the identical shape, so the *structure* is trustworthy even though the OpenClaw pages aren't.
- **[marketing]** OpenClaw docs (`docs.openclaw.ai/clawhub/skill-format`, `/tools/skills`) describe a six-tier precedence loader (workspace → project → personal → managed → bundled → plugin), skill identity from the `name` field not the folder, and an optional `skill.json` for "fine-grained tool permissions." Plausible and internally consistent, but vendor/SEO-grade; do not treat field names as canonical.
- **[marketing]** `win4r/openclaw-a2a-gateway` claims a v0.3.0 implementation: agent card at `/.well-known/agent-card.json`, three transports (JSON-RPC → REST → gRPC with fallback), all three Part types (Text/File/Data), bearer auth with token rotation, "bio-inspired routing." The protocol claims line up with verified A2A; the routing/scale claims read as embellishment.
- **[inferred]** Net: the skill model is **MCP-adjacent but simpler** (instructions, not a tool server), and A2A is **orthogonal** to it — A2A moves *messages between agents*; skills are *what one agent knows how to do*.

## Skill manifest shape
A skill is a **directory containing `SKILL.md`** = YAML frontmatter + markdown instruction body, discovered by scanning configured roots for any `SKILL.md`. Frontmatter (per docs, **[marketing]** but matching the real Agent-Skills convention):

```yaml
---
name: todoist-cli              # required — also the skill's identity
description: Manage Todoist tasks from the CLI.   # required — used for selection/search
version: 1.2.0
metadata:
  openclaw:                    # aliases: clawdbot / clawdis
    requires:
      env:  [TODOIST_API_KEY]  # env that must be present
      bins: [curl]             # binaries that must all exist
      anyBins: [...]           # at least one must exist
    primaryEnv: TODOIST_API_KEY
    envVars: [{name, required, description}]
    os: [darwin, linux]
    emoji: "✅"
    homepage: https://...
---
<markdown body: how/when the agent should act>
```

Optional `skill.json` registers tool permissions with the runtime. **Takeaway:** this is essentially our `job-descriptions/*.md` already — runtime-discovered markdown capability units. We are convergent with this design, not behind it.

## A2A contract (the verified part)
- **Discovery:** GET an **Agent Card** at `/.well-known/agent.json` (the gateway also serves `/.well-known/agent-card.json`). Card = `{ name, description, url, skills:[{id,name,description}], capabilities, securitySchemes }`. Also discoverable via registries or direct config.
- **Capability advertisement:** the `skills[]` array on the card; peers route by matching skill ids/tags.
- **Messaging:** JSON-RPC 2.0 over HTTP(S); methods send a message containing one or more **Parts** — `TextPart`, `FilePart` (URI or base64), `DataPart` (structured JSON). Long-running work streams task-status updates over **SSE**.
- **Auth:** bearer token per peer (declared in `securitySchemes`); the gateway adds multi-token rotation.

## Recommendation for Buster Claw
**Default: stay on the durable pull-queue (`Dispatch`) + loopback CLI. Do not adopt A2A now.** Reasons:
1. **A2A solves a problem we don't have yet.** It is a *cross-process, cross-machine, cross-vendor* RPC for agents that don't share state. Our roles/sub-agents share a workspace, a SQLite store, and the Sentinel feed. In-process + durable-queue coordination is strictly simpler and already gives us durability, replay, and audit — things A2A's fire-and-stream model would force us to rebuild.
2. **The skill half is already ours.** `job-descriptions/*.md` ≈ SKILL.md. The cheap, high-value move is to **align our manifest frontmatter to the SKILL.md convention** (name/description/requires/version) so we could later import ClawHub-style skills and stay legible to that ecosystem. Pure file work, no protocol commitment.
3. **A2A would bypass Sentinel.** An A2A inbound endpoint is a new un-audited message ingress and an outbound exfiltration path. Adding it before we have a Sentinel-aware A2A adapter is a net security regression for a file-first app.

**Adopt A2A when (and only when) ONE of these becomes true:**
- We need **cross-machine agents** (e.g. a user's roles running on two boxes, or a remote worker pool) — durable-queue-over-localhost stops being enough and we need a discovery + transport standard.
- We need to **interoperate with third-party agents** the user already runs (an OpenClaw/LangGraph/other-vendor agent) — A2A is the lingua franca and reimplementing it would be wasteful.
- A user explicitly wants to **publish a Buster Claw agent as a callable peer**.

When that trigger fires, the cheap path is a **single A2A gateway process** that (a) serves one Agent Card whose `skills[]` is generated from our `job-descriptions/*.md`, and (b) translates inbound A2A messages into `Dispatch` enqueues and streams results back over SSE — i.e. A2A as an *edge adapter on top of the queue*, never a replacement for it, with Sentinel sitting on that edge.
