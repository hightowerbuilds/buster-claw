# Architecture Notes

Buster Claw is a Wails desktop application with a Go backend and SolidJS frontend.

## Runtime Shape

- `main.go` starts the Wails application.
- `app.go` exposes methods to the frontend through the Wails binding root.
- `internal/**` contains backend domain packages for ingestion, orchestration, providers, scheduling, hooks, webhooks, delivery, MCP, memory, and library storage.
- `frontend/src/**` contains the SolidJS application.
- `frontend/wailsjs/**` is generated by Wails and should not be hand-edited.

## Runtime Data

The app persists local state beside the app runtime. Current files and directories include:

- `sources.json`: configured ingestion sources.
- `providers.json`: configured model providers and active provider selection.
- `mcp.json`: configured MCP servers.
- `hooks.json`: reactive hook definitions.
- `delivery.json`: report delivery destinations.
- `webhooks.json`: webhook definitions.
- `Library/raw/`: fetched source documents.
- `Library/reports/`: generated markdown reports and `manifest.json`.
- `Library/queue.json`: persisted analysis queue.
- `memory/pneuma.md`: durable local memory.

Some of these files may not exist until the related feature is configured or run for the first time.

## Modularization Direction

The public Wails API should remain stable while implementation moves into smaller feature-owned files and backend services. The target shape is:

- Keep `App` as a thin Wails facade.
- Move domain behavior into testable packages under `internal/`.
- Keep frontend views, API calls, stores, and styles separated by feature.
- Keep generated Wails bindings out of manual refactors.
